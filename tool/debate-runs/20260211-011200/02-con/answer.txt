`2026-02-11 01:12 (GMT+9)` 기준 실전 운영안은 아래처럼 고정하면 됩니다.

1. 운영 모드 2개
- `Fast`: 지연 최소화 (`pro/con` 짧은 timeout, retry 1회)
- `Safe`: 안정성 우선 (timeout 상향, retry 2회, fallback synthesis 강제)

2. 단계별 정책
- `pro/con`: 병렬 유지, 실패 시 짧은 형식 프롬프트로 1회 재시도
- `synthesis`: 한쪽 입력만 있어도 축약 합성 수행
- 완전 실패 시: 최소 결론 템플릿 반환(서비스 공백 방지)

3. 재시도 표준
- 고정 지연 금지, `exponential backoff + jitter`
- 재시도 대상: `timeout` 중심
- 포맷 오류: 프롬프트 축약 후 1회만

4. 관측 지표(필수)
- `p50/p95 end-to-end latency`
- `timeout rate`
- `retry success rate`
- `blocked_missing_inputs rate`

5. 운영 자동화
- 연속 실패 임계치 초과 시 `degraded mode` 전환
- 특정 에이전트 불안정 시 단일 에이전트 우회
- 일 단위 추이 보고서로 파라미터 자동 튜닝 후보 산출

6. 배포 방식
- 파라미터 변경은 A/B(20%→50%→100%)
- 승격 조건: `blocked rate` 개선 + `p95` 악화 허용치 이내

원하면 다음으로 `Fast/Safe` 각각의 권장 기본 파라미터 값(초 단위)까지 바로 고정해드리겠습니다.
