핵심은 이미 코드에서 병렬 실행을 하고 있는데, 실제 병목이 `병렬성`이 아니라 `타임아웃/재시도 설계`에 있다는 점입니다.

- 근거: `tool/debate-runs/20260211-005104/timeline.log`에서 `01-pro`, `02-con`, `01b-pro-retry`, `02b-con-retry`가 모두 `timeout`으로 끝나고 `blocked_missing_inputs`로 종료됨.
- 현재 요청도 `2026-02-11 00:53`에 `tool/debate-runs/20260211-005352/timeline.log`에서 시작만 기록된 상태라, 같은 패턴 재발 가능성이 큼.

속도 최적화 우선순위(성능 유지 기준):

1. 타임아웃 정책 분리
- `tool/ai_debate.ps1`의 `StepTimeoutSec`를 단계별로 다르게.
- `pro/con`: 짧게(예: 15~25s), `retry`: 더 짧게(예: 8~12s), `synthesis`: 중간(예: 20~30s).
- 지금처럼 동일 타임아웃이면 실패 시 전체 런 지연이 크게 누적됨.

2. 재시도 전략 개선
- 고정 지연(`RetryDelaySec`) 대신 짧은 지수 백오프 + jitter.
- 예: `0.5s -> 1s`(+랜덤 100~300ms), 최대 2회.
- 실패 시 즉시 같은 프롬프트 재전송보다 “초간결 포맷 강제 프롬프트”로 전환.

3. 프로세스 스폰 오버헤드 축소
- `Start-Job`(별도 PowerShell 프로세스) 대신 `ThreadJob`/Runspace Pool 전환.
- 현재 `Invoke-AgentTurn`, `Invoke-AgentTurnsParallel`에서 매 턴 프로세스를 띄우는 구조가 tail latency를 키움 (`tool/ai_debate.ps1`).

4. 실패 허용(fallback) 경로 추가
- `pro` 또는 `con` 한쪽만 성공해도 synthesis를 “축약 모드”로 진행.
- 지금은 둘 다 성공해야 해서 `blocked_missing_inputs` 비율이 높아짐.

5. 파싱 실패를 속도 저하로 연결하지 않기
- 과거 `no_json` 이력이 있으므로 JSON 파싱 실패 시 즉시 `plain_text` 수용(재시도 최소화).
- `Convert-RawToResult` 경로를 “엄격 JSON 우선”이 아니라 “텍스트 우선, JSON 있으면 usage 추출”로 변경.

운영 지표(필수):
- `total_latency_p50/p95`
- `timeout_rate` (`parse=timeout`)
- `blocked_rate` (`parse=blocked_missing_inputs`)
- `retry_success_rate`
- 이 4개를 `summary.json`에서 집계해서 튜닝 전/후 비교.
